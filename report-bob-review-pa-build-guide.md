### 2.3 Phase 3 — Exercise Content Generation (Core of the Guide)

This was the most intensive phase. For each of the 20 exercises:

1. A short context prompt was given to Bob, specifying the domain, the PA task, and the 24Retail model context.
2. Bob generated the full exercise, including:
   - The exact prompt to run (ready to copy into Bob)
   - The step-by-step interaction flow
   - A reference result (TI code, MDX query, TM1 rules, architecture diagram, migration plan, etc.)
   - With Bob / Without Bob comparison
   - Discussion points for facilitators

For the exercises that required live server data (e.g., Exercise 1.1 — Reverse-engineering, Exercise 3.1 — KPI rules, Exercise 4.1 — SAP mapping), Bob was run **directly against the 24Retail server via MCP** during authoring. The outputs returned by Bob's MCP tool calls were captured and embedded directly into the guide as reference results. This ensured that all reference outputs are grounded in real server state, not fabricated examples.

**Selected examples of what Bob produced live via MCP:**
- A complete dimensional inventory of 34 cubes and 60+ TI processes from the 24Retail server
- Full TI process code for ETL load, seasonal spread, multi-cube aggregation, and SAP mapping
- TM1 rules for Contribution Margin KPIs, currency conversion, and Budget vs Actual variance
- Anomaly detection using `perform_outlier_detection` on cube views
- Deployment checklists and migration plans derived from real metadata

> **Without Bob:** Each exercise would require a senior PA consultant, TM1 Architect access, manual coding, and documentation. Average effort: 2–4 hours per exercise × 20 exercises = **40–80 hours**.

### 3.3 BobCoins Optimisation Tips Applied

The following practices were applied to stay within the coin budget:

1. **Batch prompting** — Multiple exercise sections were generated in a single structured prompt rather than one prompt per section.
2. **Template reuse** — The 8-section exercise template was defined once; Bob applied it consistently for all 20 exercises without re-specification.
3. **MCP `alwaysAllow` configuration** — Frequently called MCP tools were listed in `alwaysAllow` to eliminate confirmation overhead and reduce turn count.
4. **Context compression** — Each generation prompt referenced the established structure rather than re-describing the full guide, minimising context size.
5. **Live captures over manual writes** — Where Bob produced a live MCP result, it was embedded directly rather than re-prompting Bob to write a simulated version.
